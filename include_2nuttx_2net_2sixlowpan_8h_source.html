68 /* Fragment header.
69  *
70  * The fragment header is used when the payload is too large to fit in a
71  * single radio frame. The fragment header contains three fields: Datagram
72  * size, datagram tag and datagram offset.
73  *
74  * 1. Datagram size describes the total (un-fragmented) payload.
75  * 2. Datagram tag identifies the set of fragments and is used to match
76  * fragments of the same payload.
77  * 3. Datagram offset identifies the fragment’s offset within the un-
78  * fragmented payload.
79  *
80  * The fragment header length is 4 bytes for the first header and 5
81  * bytes for all subsequent headers.
82  */

441 /****************************************************************************
442  * Name: sixlowpan_input
443  *
444  * Description:
445  * Process an incoming 6LoWPAN frame.
446  *
447  * This function is called when the radio device driver has received an
448  * frame from the network. The frame from the device driver must be
449  * provided in by the IOB frame argument of the function call:
450  *
451  * - The frame data is in the IOB io_data[] buffer,
452  * - The length of the frame is in the IOB io_len field, and
453  * - The offset past and radio MAC header is provided in the io_offset
454  * field.
455  *
456  * The frame argument may refer to a single frame (a list of length one)
457  * or may it be the head of a list of multiple frames.
458  *
459  * - The io_flink field points to the next frame in the list (if enable)
460  * - The last frame in the list will have io_flink == NULL.
461  *
462  * An non-NULL d_buf of size CONFIG_NET_6LOWPAN_MTU + CONFIG_NET_GUARDSIZE
463  * must also be provided. The frame will be decompressed and placed in
464  * the d_buf. Fragmented packets will also be reassembled in the d_buf as
465  * they are received (meaning for the driver, that two packet buffers are
466  * required: One for reassembly of RX packets and one used for TX polling).
467  *
468  * After each frame is processed into d_buf, the IOB is deallocated. If
469  * reassembly is incomplete, the partially reassembled packet must be
470  * preserved by the radio network drvier and provided again when the next
471  * frame is received.
472  *
473  * When the packet in the d_buf is fully reassembled, it will be provided
474  * to the network as with any other received packet. d_len will be set
475  * the length of the uncompressed, reassembled packet.
476  *
477  * After the network processes the packet, d_len will be set to zero.
478  * Network logic may also decide to send a response to the packet. In
479  * that case, the outgoing network packet will be placed in d_buf and
480  * d_len will be set to a non-zero value. That case is handled by this
481  * function.
482  *
483  * If that case occurs, the packet will be converted to a list of
484  * compressed and possibly fragmented frames and provided to the MAC
485  * network driver via the req_data() method as with other TX operations.
486  *
487  * Input Parameters:
488  * radio The radio network driver interface.
489  * framelist - The head of an incoming list of frames. Normally this
490  * would be a single frame. A list may be provided if
491  * appropriate, however.
492  * metadata - Meta data characterizing the received packet. The specific
493  * type of this metadata is obfuscated and depends on the
494  * type of the radio driver. This could be be either
495  * (1) struct ieee802154_data_ind_s for an IEEE 802.15.4
496  * radio, or (2) struct pktradio_metadata_s for a non-standard
497  * packet radio.
498  *
499  * If there are multilple frames in the list, this metadata
500  * must apply to all of the frames in the list.
501  *
502  * Returned Value:
503  * Ok is returned on success; Othewise a negated errno value is returned.
504  *
505  ****************************************************************************/
