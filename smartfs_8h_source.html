56 /* SMART Definitions ********************************************************/
57 /* General SMART organization. The following example assumes 4 logical
58  * sectors per FLASH erase block. The actual relationship is determined by
59  * the FLASH geometry reported by the MTD driver.
60  *
61  * ERASE LOGICAL Sectors begin with a sector header. Sectors may
62  * BLOCK SECTOR CONTENTS be marked as "released," pending garbage collection
63  * n 4*n --+---------------+
64  * Hdr |LLLLLLLLLLLLLLL| Logical sector number (2 bytes)
65  * |QQQQQQQQQQQQQQQ| Sequence number (2 bytes)
66  * |SSSSSSSSSSSSSSS| Status bits (1 byte)
67  * +---------------+
68  * |TTTTTTTTTTTTTTT| Sector Type (dir or file) (1 byte)
69  * |NNNNNNNNNNNNNNN| Number of next logical sector in chain
70  * |UUUUUUUUUUUUUUU| Number of bytes used in this sector
71  * | |
72  * | (Sector Data) |
73  * | |
74  * 4*n+1 --+---------------+
75  * Sector Hdr |LLLLLLLLLLLLLLL| Logical sector number (2 bytes)
76  * |QQQQQQQQQQQQQQQ| Sequence number (2 bytes)
77  * |SSSSSSSSSSSSSSS| Status bits (1 byte)
78  * +---------------+
79  * FS Hdr |TTTTTTTTTTTTTTT| Sector Type (dir or file) (1 byte)
80  * |NNNNNNNNNNNNNNN| Number of next logical sector in chain
81  * |UUUUUUUUUUUUUUU| Number of bytes used in this sector
82  * | |
83  * | (Sector Data) |
84  * | |
85  * 4*n+2 --+---------------+
86  * Sector Hdr |LLLLLLLLLLLLLLL| Logical sector number (2 bytes)
87  * |QQQQQQQQQQQQQQQ| Sequence number (2 bytes)
88  * |SSSSSSSSSSSSSSS| Status bits (1 byte)
89  * +---------------+
90  * FS Hdr |TTTTTTTTTTTTTTT| Sector Type (dir or file) (1 byte)
91  * |NNNNNNNNNNNNNNN| Number of next logical sector in chain
92  * |UUUUUUUUUUUUUUU| Number of bytes used in this sector
93  * | |
94  * | (Sector Data) |
95  * | |
96  * 4*n+3 --+---------------+
97  * Sector Hdr |LLLLLLLLLLLLLLL| Logical sector number (2 bytes)
98  * |QQQQQQQQQQQQQQQ| Sequence number (2 bytes)
99  * |SSSSSSSSSSSSSSS| Status bits (1 byte)
100  * +---------------+
101  * FS Hdr |TTTTTTTTTTTTTTT| Sector Type (dir or file) (1 byte)
102  * |NNNNNNNNNNNNNNN| Number of next logical sector in chain
103  * |UUUUUUUUUUUUUUU| Number of bytes used in this sector
104  * | |
105  * | (Sector Data) |
106  * | |
107  * n+1 4*(n+1) --+---------------+
108  * Sector Hdr |LLLLLLLLLLLLLLL| Logical sector number (2 bytes)
109  * |QQQQQQQQQQQQQQQ| Sequence number (2 bytes)
110  * |SSSSSSSSSSSSSSS| Status bits (1 byte)
111  * +---------------+
112  * FS Hdr |TTTTTTTTTTTTTTT| Sector Type (dir or file) (1 byte)
113  * |NNNNNNNNNNNNNNN| Number of next logical sector in chain
114  * |UUUUUUUUUUUUUUU| Number of bytes used in this sector
115  * | |
116  * | |
117  * | |
118  * --+---------------+
119  *
120  * General operation:
121  * Physical sectors are allocated and assigned a logical sector number
122  * and a starting sequence number of zero.
123  *
124  * SECTOR HEADER:
125  * The sector header (first 5 bytes) tracks the state of each sector and
126  * is used by the SMART MTD block driver. At the block level, there is
127  * no notion of sector chaining, only allocated sectors within erase
128  * blocks.
129  *
130  * FILE SYSTEM (FS) HEADER:
131  * The file system header (next 5 bytes) tracks file and directory entries
132  * and chains.
133  *
134  * SMART Limitations:
135  * 1. SMART currently depends on the underlying MTD block driver supporting
136  * single-byte programming operation. This is due to the method it
137  * uses for marking a sector as "released", committed, etc.
138  * 2. Garbage collection can occur when a new sector is allocated or when
139  * existing sector data is overwritten with new data. Thus, occasionally,
140  * file writing may take longer than other times.
141  * 3. The implementation curently does not track bad blocks on the device.
142  * 4. There is no true wear-leveling implemented yet, though provesion have
143  * been made to reserve logical sectors to allow it to be added using
144  * a "sector aging" tracking mechanism.
145  */
146 
147 /* Values for SMART inode state.
148  *
149  * SMART_STATE_FILE - The inode is a valid usuable, file
150  * INODE_STATE_DELETED - The inode has been deleted.
151  * Other values - The inode is bad and has an invalid state.
152  *
153  * Care is taken so that the VALID to DELETED transition only involves burning
154  * bits from the erased to non-erased state.
