50 /* The dlfcn.h header defines at least the following macros for use in the
51  * construction of a dlopen() mode argument:
52  *
53  * RTLD_LAZY - Relocations are performed at an implementation-dependent
54  * time, ranging from the time of the dlopen() call until
55  * the first reference to a given symbol occurs. Specifying
56  * RTLD_LAZY should improve performance on implementations
57  * supporting dynamic symbol binding as a process may not
58  * reference all of the functions in any given object. And,
59  * for systems supporting dynamic symbol resolution for
60  * normal process execution, this behaviour mimics the
61  * normal handling of process execution.
62  * RTLD_NOW - All necessary relocations are performed when the object
63  * is first loaded. This may waste some processing if
64  * relocations are performed for functions that are never
65  * referenced. This behaviour may be useful for
66  * applications that need to know as soon as an object is
67  * loaded that all symbols referenced during execution will
68  * be available.
69  *
70  * Any object loaded by dlopen() that requires relocations against global
71  * symbols can reference the symbols in the original process image file,
72  * any objects loaded at program startup, from the object itself as well as
73  * any other object included in the same dlopen() invocation, and any
74  * objects that were loaded in any dlopen() invocation and which specified
75  * the RTLD_GLOBAL flag. To determine the scope of visibility for the
76  * symbols loaded with a dlopen() invocation, the mode parameter should be
77  * bitwise or'ed with one of the following values:
78  *
79  * RTLD_GLOBAL - The object's symbols are made available for the
80  * relocation processing of any other object. In addition,
81  * symbol lookup using dlopen(0, mode) and an associated
82  * dlsym() allows objects loaded with this mode to be
83  * searched.
84  * RTLD_LOCAL - All symbols are not made available for relocation
85  * processing by other modules.
86  *
87  * Reference: OpenGroup.org
88  */
89 
90 #define RTLD_LAZY (0 << 0)
91 #define RTLD_NOW (1 << 0)
92 #define RTLD_GLOBAL (1 << 1)
93 #define RTLD_LOCAL (1 << 2)

127 /****************************************************************************
128  * Name: dlopen
129  *
130  * Description:
131  * dlopen() makes an executable object file specified by file available to
132  * the calling program. The class of files eligible for this operation and
133  * the manner of their construction are specified by the implementation,
134  * though typically such files are executable objects such as shared
135  * libraries, relocatable files or programs. Note that some implementations
136  * permit the construction of dependencies between such objects that are
137  * embedded within files. In such cases, a dlopen() operation will load
138  * such dependencies in addition to the object referenced by file.
139  * Implementations may also impose specific constraints on the construction
140  * of programs that can employ dlopen() and its related services.
141  *
142  * If a file is specified in multiple dlopen() invocations, mode is
143  * interpreted at each invocation. Note, however, that once RTLD_NOW has
144  * been specified all relocations will have been completed rendering
145  * further RTLD_NOW operations redundant and any further RTLD_LAZY
146  * operations irrelevant. Similarly note that once RTLD_GLOBAL has been
147  * specified the object will maintain the RTLD_GLOBAL status regardless
148  * of any previous or future specification of RTLD_LOCAL, so long as the
149  * object remains in the address space (see dlclose()).
150  *
151  * Symbols introduced into a program through calls to dlopen() may be
152  * used in relocation activities. Symbols so introduced may duplicate
153  * symbols already defined by the program or previous dlopen()
154  * operations. To resolve the ambiguities such a situation might
155  * present, the resolution of a symbol reference to symbol definition is
156  * based on a symbol resolution order. Two such resolution orders are
157  * defined: load or dependency ordering. Load order establishes an
158  * ordering among symbol definitions, such that the definition first
159  * loaded (including definitions from the image file and any dependent
160  * objects loaded with it) has priority over objects added later (via
161  * dlopen()). Load ordering is used in relocation processing. Dependency
162  * ordering uses a breadth-first order starting with a given object,
163  * then all of its dependencies, then any dependents of those, iterating
164  * until all dependencies are satisfied. With the exception of the global
165  * symbol object obtained via a dlopen() operation on a file of 0,
166  * dependency ordering is used by the dlsym() function. Load ordering is
167  * used in dlsym() operations upon the global symbol object.
168  *
169  * When an object is first made accessible via dlopen() it and its
170  * dependent objects are added in dependency order. Once all the objects
171  * are added, relocations are performed using load order. Note that if an
172  * object or its dependencies had been previously loaded, the load and
173  * dependency orders may yield different resolutions.
174  *
175  * The symbols introduced by dlopen() operations, and available through
176  * dlsym() are at a minimum those which are exported as symbols of global
177  * scope by the object. Typically such symbols will be those that were
178  * specified in (for example) C source code as having extern linkage. The
179  * precise manner in which an implementation constructs the set of
180  * exported symbols for a dlopen() object is specified by that
181  * implementation.
182  *
183  * Input Parameters:
184  * file - Used to construct a pathname to the object file. If file
185  * contains a slash character, the file argument is used as the
186  * pathname for the file. Otherwise, file is used in an
187  * implementation-dependent manner to yield a pathname.
188  *
189  * If the value of file is 0, dlopen() provides a handle on a
190  * global symbol object. This object provides access to the symbols
191  * from an ordered set of objects consisting of the original
192  * program image file, together with any objects loaded at program
193  * startup as specified by that process image file (for example,
194  * shared libraries), and the set of objects loaded using a
195  * dlopen() operation together with the RTLD_GLOBAL flag. As the
196  * latter set of objects can change during execution, the set
197  * identified by handle can also change dynamically.
198  *
199  * Only a single copy of an object file is brought into the address
200  * space, even if dlopen() is invoked multiple times in reference
201  * to the file, and even if different pathnames are used to
202  * reference the file.
203  * mode - Describes how dlopen() will operate upon file with respect to
204  * the processing of relocations and the scope of visibility of the
205  * symbols provided within file. When an object is brought into the
206  * address space of a process, it may contain references to symbols
207  * whose addresses are not known until the object is loaded. These
208  * references must be relocated before the symbols can be accessed.
209  * The mode parameter governs when these relocations take place.
210  * See definitions above for values of the mode parameter:.
211  *
212  * Returned Value:
213  * A successful dlopen() returns a handle which the caller may use on
214  * subsequent calls to dlsym() and dlclose(). The value of this handle
215  * should not be interpreted in any way by the caller.
216  *
217  * If file cannot be found, cannot be opened for reading, is not of an
218  * appropriate object format for processing by dlopen(), or if an error
219  * occurs during the process of loading file or relocating its symbolic
220  * references, dlopen() will return NULL. More detailed diagnostic
221  * information will be available through dlerror().
222  *
223  * Reference: OpenGroup.org
224  *
225  * ****************************************************************************/
226 
227 FAR void *dlopen(FAR const char *file, int mode);

229 /****************************************************************************
230  * Name: dlsym
231  *
232  * Description:
233  * dlsym() allows a process to obtain the address of a symbol defined
234  * within an object made accessible through a dlopen() call. handle is the
235  * value returned from a call to dlopen() (and which has not since been
236  * released via a call to dlclose()), name is the symbol's name as a
237  * character string.
238  *
239  * dlsym() will search for the named symbol in all objects loaded
240  * automatically as a result of loading the object referenced by handle
241  * (see dlopen()). Load ordering is used in dlsym() operations upon the
242  * global symbol object. The symbol resolution algorithm used will be
243  * dependency order as described in dlopen().
244  *
245  * Input Parameters:
246  * handle - The opaque, non-NULL value returned by a previous successful
247  * call to dlopen().
248  * name - A pointer to the symbol name string.
249  *
250  * Returned Value:
251  * If handle does not refer to a valid object opened by dlopen(), or if
252  * the named symbol cannot be found within any of the objects associated
253  * with handle, dlsym() will return NULL. More detailed diagnostic
254  * information will be available through dlerror().
255  *
256  * Reference: OpenGroup.org
257  *
258  ****************************************************************************/
259 
260 FAR void *dlsym(FAR void *handle, FAR const char *name);

262 /****************************************************************************
263  * Name: dlclose
264  *
265  * Description:
266  * dlclose() is used to inform the system that the object referenced by a
267  * handle returned from a previous dlopen() invocation is no longer needed
268  * by the application.
269  *
270  * The use of dlclose() reflects a statement of intent on the part of the
271  * process, but does not create any requirement upon the implementation,
272  * such as removal of the code or symbols referenced by handle. Once an
273  * object has been closed using dlclose() an application should assume
274  * that its symbols are no longer available to dlsym(). All objects loaded
275  * automatically as a result of invoking dlopen() on the referenced object
276  * are also closed.
277  *
278  * Although a dlclose() operation is not required to remove structures
279  * from an address space, neither is an implementation prohibited from
280  * doing so. The only restriction on such a removal is that no object will
281  * be removed to which references have been relocated, until or unless all
282  * such references are removed. For instance, an object that had been
283  * loaded with a dlopen() operation specifying the RTLD_GLOBAL flag might
284  * provide a target for dynamic relocations performed in the processing of
285  * other objects - in such environments, an application may assume that no
286  * relocation, once made, will be undone or remade unless the object
287  * requiring the relocation has itself been removed.
288  *
289  * Input Parameters:
290  * handle - The opaque, non-NULL value returned by a previous successful
291  * call to dlopen().
292  *
293  * Returned Value:
294  * If the referenced object was successfully closed, dlclose() returns 0.
295  * If the object could not be closed, or if handle does not refer to an
296  * open object, dlclose() returns a non-zero value. More detailed
297  * diagnostic information will be available through dlerror().
298  *
299  * Reference: OpenGroup.org
300  * ****************************************************************************/
301 
302 int dlclose(FAR void *handle);
