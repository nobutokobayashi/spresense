891 /* Level 2 Page table start addresses.
892  *
893  * The maximum size of the L1 page table is:
894  *
895  * (4GB address range / 1 MB per section ) * 4 bytes per entry = 16KB
896  *
897  * The maximum size of the L2 page table is:
898  *
899  * (4GB address range / 4 KB per page ) * 4 bytes per entry = 4MB
900  *
901  * 16KB of memory is reserved hold the page table for the virtual mappings. A
902  * portion of this table is not accessible in the virtual address space (for
903  * normal operation with a one-to-one address mapping). There is this large
904  * hole in the physcal address space for which there will never be level 1
905  * mappings:
906  *
907  * 0x80000000-0xefffffff: Undefined (1.75 GB)
908  *
909  * That is the offset where the main L2 page tables will be positioned. This
910  * corresponds to page table offsets 0x00002000 up to 0x00003c00. That
911  * is 1792 entries, each mapping 4KB of address for a total of 7MB of virtual
912  * address space)
913  *
914  * Up to two L2 page tables may be used:
915  *
916  * 1) One mapping the vector table. However, L2 page tables must be aligned
917  * to 1KB address boundaries, so the minimum L2 page table size is then
918  * 1KB, mapping up a full megabyte of virtual address space.
919  *
920  * This L2 page table is only allocated if CONFIG_ARCH_LOWVECTORS is *not*
921  * defined. The i.MX6 boot-up logic will map the beginning of the boot
922  * memory to address 0x0000:0000 using both the MMU and the AXI matrix
923  * REMAP register. So no L2 page table is required.
924  *
925  * 2) If on-demand paging is supported (CONFIG_PAGING=y), than an additional
926  * L2 page table is needed. This page table will use the remainder of
927  * the address space.
928  */
