246 /****************************************************************************
247  * Name: pg_miss
248  *
249  * Description:
250  * This function is called from architecture-specific memory segmentation
251  * fault handling logic. This function will perform the following
252  * operations:
253  *
254  * 1) Sanity checking.
255  * - ASSERT if the currently executing task is the page fill worker
256  * thread. The page fill worker thread is how the page fault
257  * is resolved and all logic associated with the page fill worker
258  * must be "locked" and always present in memory.
259  * 2) Block the currently executing task.
260  * - Call up_block_task() to block the task at the head of the ready-
261  * to-run list. This should cause an interrupt level context switch
262  * to the next highest priority task.
263  * - The blocked task will be marked with state TSTATE_WAIT_PAGEFILL
264  * and will be retained in the g_waitingforfill prioritized task
265  * list.
266  * 3) Boost the page fill worker thread priority.
267  * - Check the priority of the task at the head of the g_waitingforfill
268  * list. If the priority of that task is higher than the current
269  * priority of the page fill worker thread, then boost the priority
270  * of the page fill worker thread to that priority.
271  * 4) Signal the page fill worker thread.
272  * - Is there a page fill pending? If not then signal the worker
273  * thread to start working on the queued page fill requests.
274  *
275  * Input Parameters:
276  * None - The head of the ready-to-run list is assumed to be task that
277  * caused the exception.
278  *
279  * Returned Value:
280  * None - Either this function function succeeds or an assertion occurs.
281  *
282  * Assumptions:
283  * - It is assumed that this function is called from the level of an
284  * exception handler and that all interrupts are disabled.
285  * - It is assumed that currently executing task (the one at the head of
286  * the ready-to-run list) is the one that cause the fault. This will
287  * always be true unless the page fault occurred in an interrupt handler.
288  * Interrupt handling logic must always be present and "locked" into
289  * memory.
290  * - The chip-specific page fault exception handler has already verified
291  * that the exception did not occur from interrupt/exception handling
292  * logic.
293  * - As mentioned above, the task causing the page fault must not be the
294  * page fill worker thread because that is the only way to complete the
295  * page fill.
296  *
297  * NOTES:
298  * 1. One way to accomplish this would be a two pass link phase:
299  * - In the first phase, create a partially linked objected containing
300  * all interrupt/exception handling logic, the page fill worker thread
301  * plus all parts of the IDLE thread (which must always be available
302  * for execution).
303  * - All of the .text and .rodata sections of this partial link should
304  * be collected into a single section.
305  * - The second link would link the partially linked object along with
306  * the remaining object to produce the final binary. The linker
307  * script should position the "special" section so that it lies
308  * in a reserved, "non-swappable" region.
309  *
310  ****************************************************************************/
311 
312 void pg_miss(void);

351 /****************************************************************************
352  * Name: up_allocpage()
353  *
354  * Description:
355  * This architecture-specific function will set aside page in memory and map
356  * the page to its correct virtual address. Architecture-specific context
357  * information saved within the TCB will provide the function with the
358  * information needed to identify the virtual miss address.
359  *
360  * This function will return the allocated physical page address in vpage.
361  * The size of the underlying physical page is determined by the
362  * configuration setting CONFIG_PAGING_PAGESIZE.
363  *
364  * NOTE 1: This function must always return a page allocation. If all
365  * available pages are in-use (the typical case), then this function will
366  * select a page in-use, un-map it, and make it available.
367  *
368  * NOTE 2: If an in-use page is un-mapped, it may be necessary to flush the
369  * instruction cache in some architectures.
370  *
371  * NOTE 3: Allocating and filling a page is a two step process. up_allocpage()
372  * allocates the page, and up_fillpage() fills it with data from some non-
373  * volatile storage device. This distinction is made because up_allocpage()
374  * can probably be implemented in board-independent logic whereas up_fillpage()
375  * probably must be implemented as board-specific logic.
376  *
377  * NOTE 4: The initial mapping of vpage should be read-able and write-
378  * able (but not cached). No special actions will be required of
379  * up_fillpage() in order to write into this allocated page.
380  *
381  * Input Parameters:
382  * tcb - A reference to the task control block of the task that needs to
383  * have a page fill. Architecture-specific logic can retrieve page
384  * fault information from the architecture-specific context
385  * information in this TCB to perform the mapping.
386  *
387  * Returned Value:
388  * This function will return zero (OK) if the allocation was successful.
389  * A negated errno value may be returned if an error occurs. All errors,
390  * however, are fatal.
391  *
392  * Assumptions:
393  * - This function is called from the normal tasking context (but with
394  * interrupts disabled). The implementation must take whatever actions
395  * are necessary to assure that the operation is safe within this
396  * context.
397  *
398  ****************************************************************************/
399 
400 int up_allocpage(FAR struct tcb_s *tcb, FAR void **vpage);

402 /****************************************************************************
403  * Name: up_fillpage()
404  *
405  * Description:
406  * After a page is allocated and mapped by up_allocpage(), the actual
407  * filling of the page with data from the non-volatile, must be performed
408  * by a separate call to the architecture-specific function, up_fillpage().
409  * This function is non-blocking, it will start an asynchronous page fill.
410  * The common paging logic will provide a callback function, pg_callback,
411  * that will be called when the page fill is finished (or an error occurs).
412  * This callback is assumed to occur from an interrupt level when the
413  * device driver completes the fill operation.
414  *
415  * NOTE 1: Allocating and filling a page is a two step process. up_allocpage()
416  * allocates the page, and up_fillpage() fills it with data from some non-
417  * volatile storage device. This distinction is made because up_allocpage()
418  * can probably be implemented in board-independent logic whereas up_fillpage()
419  * probably must be implemented as board-specific logic.
420  *
421  * NOTE 2: The initial mapping of vpage will be read-able, write-able,
422  * but non-cacheable. No special actions will be required of
423  * up_fillpage() in order to write into this allocated page. If the
424  * virtual address maps to a text region, however, this function should
425  * remap the region so that is is read/execute only. It should be made
426  * cache-able in any case.
427  *
428  * Input Parameters:
429  * tcb - A reference to the task control block of the task that needs to
430  * have a page fill. Architecture-specific logic can retrieve page
431  * fault information from the architecture-specific context
432  * information in this TCB to perform the fill.
433  * pg_callbck - The function to be called when the page fill is complete.
434  *
435  * Returned Value:
436  * This function will return zero (OK) if the page fill was successfully
437  * started (the result of the page fill is passed to the callback function
438  * as the result argument). A negated errno value may be returned if an
439  * error occurs. All errors, however, are fatal.
440  *
441  * NOTE: -EBUSY has a special meaning. It is used internally to mean that
442  * the callback function has not executed. Therefore, -EBUSY should
443  * never be provided in the result argument of pg_callback.
444  *
445  * Assumptions:
446  * - This function is called from the normal tasking context (but
447  * interrupts siabled). The implementation must take whatever actions
448  * are necessary to assure that the operation is safe within this context.
449  * - Upon return, the caller will sleep waiting for the page fill callback
450  * to occur. The callback function will perform the wakeup.
451  *
452  ****************************************************************************/
