64 /* Configuration ********************************************************************/
65 /* CONFIG_CAN - Enables CAN support (MCU-specific selections are also required. For
66  * STM32, as an example, one or both of CONFIG_STM32_CAN1 or CONFIG_STM32_CAN2
67  * must also be defined)
68  * CONFIG_CAN_EXTID - Enables support for the 29-bit extended ID. Default
69  * Standard 11-bit IDs.
70  * CONFIG_CAN_FD - Enable support for CAN FD mode. For the upper half driver, this
71  * just means handling encoded DLC values (for values of DLC > 9).
72  * CONFIG_CAN_FIFOSIZE - The size of the circular buffer of CAN messages.
73  * Default: 8
74  * CONFIG_CAN_NPENDINGRTR - The size of the list of pending RTR requests.
75  * Default: 4
76  * CONFIG_CAN_LOOPBACK - A CAN driver may or may not support a loopback
77  * mode for testing. If the driver does support loopback mode, the setting
78  * will enable it. (If the driver does not, this setting will have no effect).
79  * The loopback mode may be changed later by ioctl() if the driver supports the
80  * CANIOC_SET_CONNMODES ioctl command.
81  * CONFIG_CAN_TXREADY - Add support for the can_txready() callback. This is needed
82  * only for CAN hardware the supports an separate H/W TX message FIFO. The call
83  * back is needed to keep the S/W FIFO and the H/W FIFO in sync. Work queue
84  * support is needed for this feature.
85  * CONFIG_CAN_TXREADY_HIPRI or CONFIG_CAN_TXREADY_LOPRI - Selects which work queue
86  * will be used for the can_txready() processing.
87  */

108 /* Ioctl Commands *******************************************************************/
109 /* Ioctl commands supported by the upper half CAN driver.
110  *
111  * CANIOC_RTR:
112  * Description: Send the remote transmission request and wait for the response.
113  * Argument: A reference to struct canioc_rtr_s
114  *
115  * Ioctl commands that may or may not be supported by the lower half CAN driver.
116  *
117  * CANIOC_ADD_STDFILTER:
118  * Description: Add an address filter for a standard 11 bit address.
119  * Argument: A reference to struct canioc_stdfilter_s
120  * Returned Value: A non-negative filter ID is returned on success.
121  * Otherwise -1 (ERROR) is returned with the errno
122  * variable set to indicate the nature of the error.
123  * Dependencies: None
124  *
125  * CANIOC_ADD_EXTFILTER:
126  * Description: Add an address filter for a extended 29 bit address.
127  * Argument: A reference to struct canioc_extfilter_s
128  * Returned Value: A non-negative filter ID is returned on success.
129  * Otherwise -1 (ERROR) is returned with the errno
130  * variable set to indicate the nature of the error.
131  * Dependencies: Requires CONFIG_CAN_EXTID=y
132  *
133  * CANIOC_DEL_STDFILTER:
134  * Description: Remove an address filter for a standard 11 bit address.
135  * Argument: The filter index previously returned by the
136  * CANIOC_ADD_STDFILTER command
137  * Returned Value: Zero (OK) is returned on success. Otherwise -1 (ERROR)
138  * is returned with the errno variable set to indicate the
139  * nature of the error.
140  * Dependencies: None
141  *
142  * CANIOC_DEL_EXTFILTER:
143  * Description: Remove an address filter for a standard 29 bit address.
144  * Argument: The filter index previously returned by the
145  * CANIOC_ADD_EXTFILTER command
146  * Returned Value: Zero (OK) is returned on success. Otherwise -1 (ERROR)
147  * is returned with the errno variable set to indicate the
148  * nature of the error.
149  * Dependencies: Requires CONFIG_CAN_EXTID=y
150  *
151  * CANIOC_GET_BITTIMING:
152  * Description: Return the current bit timing settings
153  * Argument: A pointer to a write-able instance of struct
154  * canioc_bittiming_s in which current bit timing values
155  * will be returned.
156  * Returned Value: Zero (OK) is returned on success. Otherwise -1 (ERROR)
157  * is returned with the errno variable set to indicate the
158  * nature of the error.
159  * Dependencies: None
160  *
161  * CANIOC_SET_BITTIMING:
162  * Description: Set new current bit timing values
163  * Argument: A pointer to a read-able instance of struct
164  * canioc_bittiming_s in which the new bit timing values
165  * are provided.
166  * Returned Value: Zero (OK) is returned on success. Otherwise -1 (ERROR)
167  * is returned with the errno variable set to indicate the
168  * nature of the error.
169  * Dependencies: None
170  *
171  * CANIOC_GET_CONNMODES:
172  * Description: Get the current bus connection modes
173  * Argument: A pointer to a write-able instance of struct
174  * canioc_connmodes_s in which the new bus modes will be returned.
175  * Returned Value: Zero (OK) is returned on success. Otherwise -1 (ERROR)
176  * is returned with the errno variable set to indicate the
177  * nature of the error.
178  * Dependencies: None
179  *
180  * CANIOC_SET_CONNMODES:
181  * Description: Set new bus connection modes values
182  * Argument: A pointer to a read-able instance of struct
183  * canioc_connmodes_s in which the new bus modes are provided.
184  * Returned Value: Zero (OK) is returned on success. Otherwise -1 (ERROR)
185  * is returned with the errno variable set to indicate the
186  * nature of the error.
187  * Dependencies: None
188  */

203 /* User defined ioctl commands are also supported. These will be forwarded
204  * by the upper-half CAN driver to the lower-half CAN driver via the co_ioctl()
205  * method fo the CAN lower-half interface. However, the lower-half driver
206  * must reserve a block of commands as follows in order prevent IOCTL
207  * command numbers from overlapping.
208  *
209  * This is generally done as follows. The first reservation for CAN driver A would
210  * look like:
211  *
212  * CAN_A_FIRST (CAN_FIRST + CAN_NCMDS) <- First command
213  * CAN_A_NCMDS 42 <- Number of commands
214  *
215  * IOCTL commands for CAN driver A would then be defined in a CAN A header file like:
216  *
217  * CANIOC_A_CMD1 _CANIOC(CAN_A_FIRST+0)
218  * CANIOC_A_CMD2 _CANIOC(CAN_A_FIRST+1)
219  * CANIOC_A_CMD3 _CANIOC(CAN_A_FIRST+2)
220  * ...
221  * CANIOC_A_CMD42 _CANIOC(CAN_A_FIRST+41)
222  *
223  * The next reservation would look like:
224  *
225  * CAN_B_FIRST (CAN_A_FIRST + CAN_A_NCMDS) <- Next command
226  * CAN_B_NCMDS 77 <- Number of commands
227  */

371 /* CAN-message Format (without Extended ID support)
372  *
373  * One based CAN-message is represented with a maximum of 10 bytes. A message is
374  * composed of at least the first 2 bytes (when there are no data bytes present).
375  *
376  * Bytes 0-1: Bits 0-3: Data Length Code (DLC)
377  * Bit 4: Remote Transmission Request (RTR)
378  * Bit 5: 1=Message ID is a bit-encoded error report (See NOTE)
379  * Bits 6-7: Unused
380  * Bytes 1-2: Bits 0-10: The 11-bit CAN identifier This message ID is a bit
381  * encoded error set if ch_error is set (See NOTE).
382  * Bits 11-15: Unused
383  * Bytes 3-10: CAN data
384  *
385  * CAN-message Format (with Extended ID support)
386  *
387  * One CAN-message consists of a maximum of 13 bytes. A message is composed of at
388  * least the first 5 bytes (when there are no data bytes).
389  *
390  * Bytes 0-3: Bits 0-28: Hold 11- or 29-bit CAN ID in host byte order. This
391  * message ID is a bit encoded error set if ch_error
392  * is set (See NOTE).
393  * Bits 29-31: Unused
394  * Byte 4: Bits 0-3: Data Length Code (DLC)
395  * Bit 4: Remote Transmission Request (RTR)
396  * Bit 5: 1=Message ID is a bit-encoded error report (See NOTE)
397  * Bit 6: Extended ID indication
398  * Bit 7: Unused
399  * Bytes 5-12: CAN data Size determined by DLC
400  *
401  * NOTE: The error indication if valid only on message reports received from the
402  * CAN driver; it is ignored on transmission. When the error bit is set, the
403  * message ID is an encoded set of error indications (see CAN_ERROR_* definitions).
404  * A more detailed report of certain errors then follows in message payload.
405  * CONFIG_CAN_ERRORS=y is required in order to receive error reports.
406  *
407  * The struct can_msg_s holds this information in a user-friendly, unpacked form.
408  * This is the form that is used at the read() and write() driver interfaces. The
409  * message structure is actually variable length -- the true length is given by
410  * the CAN_MSGLEN macro.
411  */
412

682 /************************************************************************************
683  * Name: can_txdone
684  *
685  * Description:
686  * Called when the hardware has processed the outgoing TX message. This
687  * normally means that the CAN messages was sent out on the wire. But
688  * if the CAN hardware supports a H/W TX FIFO, then this call may mean
689  * only that the CAN message has been added to the H/W FIFO. In either
690  * case, the upper-half CAN driver can remove the outgoing message from
691  * the S/W FIFO and discard it.
692  *
693  * This function may be called in different contexts, depending upon the
694  * nature of the underlying CAN hardware.
695  *
696  * 1. No H/W TX FIFO (CONFIG_CAN_TXREADY not defined)
697  *
698  * This function is only called from the CAN interrupt handler at the
699  * completion of a send operation.
700  *
701  * can_write() -> can_xmit() -> dev_send()
702  * CAN interrupt -> can_txdone()
703  *
704  * If the CAN hardware is busy, then the call to dev_send() will
705  * fail, the S/W TX FIFO will accumulate outgoing messages, and the
706  * thread calling can_write() may eventually block waiting for space in
707  * the S/W TX FIFO.
708  *
709  * When the CAN hardware completes the transfer and processes the
710  * CAN interrupt, the call to can_txdone() will make space in the S/W
711  * TX FIFO and will awaken the waiting can_write() thread.
712  *
713  * 2a. H/W TX FIFO (CONFIG_CAN_TXREADY=y) and S/W TX FIFO not full
714  *
715  * This function will be called back from dev_send() immediately when a
716  * new CAN message is added to H/W TX FIFO:
717  *
718  * can_write() -> can_xmit() -> dev_send() -> can_txdone()
719  *
720  * When the H/W TX FIFO becomes full, dev_send() will fail and
721  * can_txdone() will not be called. In this case the S/W TX FIFO will
722  * accumulate outgoing messages, and the thread calling can_write() may
723  * eventually block waiting for space in the S/W TX FIFO.
724  *
725  * 2b. H/W TX FIFO (CONFIG_CAN_TXREADY=y) and S/W TX FIFO full
726  *
727  * In this case, the thread calling can_write() is blocked waiting for
728  * space in the S/W TX FIFO. can_txdone() will be called, indirectly,
729  * from can_txready_work() running on the thread of the work queue.
730  *
731  * CAN interrupt -> can_txready() -> Schedule can_txready_work()
732  * can_txready_work() -> can_xmit() -> dev_send() -> can_txdone()
733  *
734  * The call dev_send() should not fail in this case and the subsequent
735  * call to can_txdone() will make space in the S/W TX FIFO and will
736  * awaken the waiting thread.
737  *
738  * Input Parameters:
739  * dev - The specific CAN device
740  * hdr - The 16-bit CAN header
741  * data - An array contain the CAN data.
742  *
743  * Returned Value:
744  * OK on success; a negated errno on failure.
745  *
746  * Assumptions:
747  * Interrupts are disabled. This is required by can_xmit() which is called
748  * by this function. Interrupts are explicitly disabled when called
749  * through can_write(). Interrupts are expected be disabled when called
750  * from the CAN interrupt handler.
751  *
752  ************************************************************************************/
753 
754 int can_txdone(FAR struct can_dev_s *dev);

756 /************************************************************************************
757  * Name: can_txready
758  *
759  * Description:
760  * Called from the CAN interrupt handler at the completion of a send
761  * operation. This interface is needed only for CAN hardware that
762  * supports queing of outgoing messages in a H/W FIFO.
763  *
764  * The CAN upper half driver also supports a queue of output messages in a
765  * S/W FIFO. Messages are added to that queue when when can_write() is
766  * called and removed from the queue in can_txdone() when each TX message
767  * is complete.
768  *
769  * After each message is added to the S/W FIFO, the CAN upper half driver
770  * will attempt to send the message by calling into the lower half driver.
771  * That send will not be performed if the lower half driver is busy, i.e.,
772  * if dev_txready() returns false. In that case, the number of messages in
773  * the S/W FIFO can grow. If the S/W FIFO becomes full, then can_write()
774  * will wait for space in the S/W FIFO.
775  *
776  * If the CAN hardware does not support a H/W FIFO then busy means that
777  * the hardware is actively sending the message and is guaranteed to
778  * become non-busy (i.e, dev_txready()) when the send transfer completes
779  * and can_txdone() is called. So the call to can_txdone() means that the
780  * transfer has completed and also that the hardware is ready to accept
781  * another transfer.
782  *
783  * If the CAN hardware supports a H/W FIFO, can_txdone() is not called
784  * when the tranfer is complete, but rather when the transfer is queued in
785  * the H/W FIFO. When the H/W FIFO becomes full, then dev_txready() will
786  * report false and the number of queued messages in the S/W FIFO will grow.
787  *
788  * There is no mechanism in this case to inform the upper half driver when
789  * the hardware is again available, when there is again space in the H/W
790  * FIFO. can_txdone() will not be called again. If the S/W FIFO becomes
791  * full, then the upper half driver will wait for space to become
792  * available, but there is no event to awaken it and the driver will hang.
793  *
794  * Enabling this feature adds support for the can_txready() interface.
795  * This function is called from the lower half driver's CAN interrupt
796  * handler each time a TX transfer completes. This is a sure indication
797  * that the H/W FIFO is no longer full. can_txready() will then awaken
798  * the can_write() logic and the hang condition is avoided.
799  *
800  * Input Parameters:
801  * dev - The specific CAN device
802  *
803  * Returned Value:
804  * OK on success; a negated errno on failure.
805  *
806  * Assumptions:
807  * Interrupts are disabled. This function may execute in the context of
808  * and interrupt handler.
809  *
810  ************************************************************************************/
811
