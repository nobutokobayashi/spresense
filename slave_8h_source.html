266 /* There are two interfaces defined for the implementation of SPI slave:
267  *
268  * 1) struct spi_sctrlr_s: Defines one interface between the SPI
269  * slave device and the SPI slave controller hardware. This interface
270  * is implemented by the SPI slave device controller lower-half driver
271  * and is provided to the SPI slave device driver when that driver
272  * is initialized. That SPI slave device initialization function is
273  * unique to the SPI slave implementation. The prototype is probably
274  * something like:
275  *
276  * FAR struct spi_sctrlr_s *xyz_spi_slave_initialize(int port);
277  *
278  * Given an SPI port number, this function returns an instance of the
279  * SPI slave controller interface.
280  *
281  * The actual prototype and more detailed usage instructions should
282  * appear in a header file associated with the specific SPI slave
283  * implementation.
284  *
285  * 2) struct spi_sdev_s: Defines the second interface between the SPI
286  * slave device and the SPI slave controller hardware. This interface
287  * is implemented by the SPI slave device. The slave device passes this
288  * interface to the struct spi_sctrlr_s during initialization
289  * be calling the bind() method of the struct spi_sctrlr_s
290  * interface.
291  *
292  * The basic initialization steps are:
293  *
294  * 1) Board-specific logic calls board- or chip-specific logic to create an
295  * instance of the SPI slave controller interface, struct spi_sctrlr_s.
296  *
297  * 2) Board-specific logic then calls up_dev_initialize() to initialize
298  * the SPI slave device. The board-specific logic passes the instance
299  * of struct spi_sctrlr_s to support the initialization.
300  *
301  * 3) The SPI slave device driver creates and initializes an instance of
302  * struct spi_sdev_s; it passes this instance to the bind() method of
303  * of the SPI slave controller interface.
304  *
305  * 4) The SPI slave controller will (1) call the slaved device's select()
306  * and cmddata() methods to indicate the initial state of the chip select
307  * and any command/data selection, then (2) call the slave device's
308  * getdata() method to get the value that will be shifted out the SPI
309  * clock is detected. The kind of data returned the getdata() method
310  * may be contingent on the current command/data setting reported the
311  * device cmddata() method. The driver may enqueue additional words
312  * to be shifted out at any time by The calling the SPI slave
313  * controller's enqueue() method.
314  *
315  * 5) Upon return from the bind method, the SPI slave controller will be
316  * fully "armed" and ready to begin normal SPI data transfers.
317  *
318  * A typical (non-DMA) data transfer proceeds as follows:
319  *
320  * 1) Internally, the SPI slave driver detects that the SPI chip select
321  * has gone low, selecting this device for data transfer. The SPI
322  * slave controller will notify the slave device by called its
323  * select() method.
324  *
325  * 2) If a change in the command/data state changes any time before,
326  * during, or after the chip is selected, that new command/data state
327  * will reported to the device driver via the cmddata() method.
328  *
329  * 3) As the first word is shifted in, the command or data word obtained
330  * by the initial call to getdata() will be shifted out. As soon as
331  * the clock is detected, the SPI controller driver will call the
332  * getdata() method again to get a default second word to be shifted
333  * out. NOTES: (1) the SPI slave device has only one word in bit
334  * times to provide this value! (2) The SPI device probably cannot
335  * really output anything meaning until it receives a decodes the
336  * first word received from the master.
337  *
338  * 4) When the first word from the master is shifted in, the SPI
339  * controller driver will call the device's receive() method to
340  * provide the master with the command word that was just shifted
341  * in.
342  *
343  * For the case of bi-directional data transfer or of a transfer of
344  * data from the SPI device to the master, the SPI device driver
345  * should call the controller's enqueue() method to provide the next
346  * value(s) to be shifted out. If the SPI device responds with this
347  * value before clocking begins for the next word, that that value
348  * will be used. Otherwise, the value obtained from getdata() in
349  * step 3 will be shifted out.
350  *
351  * 5) The SPI device's receive() method will be called in a similar
352  * way after each subsequent word is clocked in.
353  *
354  * For the case of bi-directional data transfer or of a uni-directional
355  * transfer of data from the SPI device to the master, the SPI device
356  * driver can call the enqueue() methods as it has new data to be shifted
357  * out. The goal of the SPI device driver for this kind of transfer is
358  * to supply valid output data at such a rate that data underruns do not
359  * occur. In the event of a data underrun, the SPI slave controller
360  * driver will fallback to the default output value obtained from the
361  * last getdata() call.
362  *
363  * The SPI device driver can detect if there is space to enqueue
364  * additional data by calling the qfull() method.
365  *
366  * For the case of uni-directional transfer of data from the master to
367  * the SPI device, there is no need to call the enqueue() method at all;
368  * the value that is shifted out is not important that fallback behavior
369  * is sufficient.
370  *
371  * The SPI slave controller driver, of course, has no sense of the
372  * directionality of a data transfer; its role is only to exchange the
373  * data shifted in from the master with new data to be shifted out from
374  * the SPI device driver.
375  *
376  * 6) The activity of 5) will continue until the master raises the chip
377  * select signal. In that case, the SPI slave controller driver will
378  * again call the SPI device's select() method. At this point, the SPI
379  * controller driver may have several words enqueued. It will not
380  * discard these unless the SPI device driver calls the qflush()
381  * method.
382  *
383  * Some master side implementations may simply tie the chip select signal
384  * to ground if there are no other devices on the SPI bus. In that case,
385  * the initial indication of chip selected will be the only call to the
386  * select() method that is made.
387  *
388  * Other SPI peripherals (such as Atmel) do not make the state of the
389  * chip select pin available (only the final rising edge transitions).
390  * So the SPI device driver implementation may use the chip select
391  * reports to optimize performance, but the design should never depend
392  * upon it.
393  *
394  * A typical DMA data transfer processes as follows:
395  * To be provided -- I do not have a design in mind to support DMA on the
396  * Slave side. The design might be very complex because:
397  *
398  * 1) You need DMA buffers of fixed size, but you cannot know the size of a
399  * transfer in advance, it could be much larger than your buffer or much
400  * smaller. The DMA would fail in either case.
401  *
402  * 2) You cannot setup the DMA before the transfer. In most SPI protocols,
403  * the first word send is a command to read or write something following
404  * by a sequence of transfers to implement the write. So you have very,
405  * very limited time window to setup the correct DMA to respond to the
406  * command. I am not certain that it can be done reliably.
407  *
408  * Inserting dummy words into the protocol between the first command word
409  * and the remaining data transfer could allow time to set up the DMA.
410  *
411  * 3) I mentioned that you do not know the size of the transfer in advance.
412  * If you set up the DMA to terminate to soon, then you lose the last part
413  * of the transfer. If you set the DMA up to be too large, then you will
414  * get no indication when the transfer completes.
415  *
416  * The chip select going high would be one possibility to detect the end
417  * of a transfer. You could cancel a DMA in progress if the CS changes,
418  * but I do not know if that would work. If there is only one device on
419  * the SPI bus, then most board designs will save a pin and simply tie CS
420  * to ground. So the CS is not always a reliable indicator of when the
421  * transfer completes.
422  *
423  * 4) The option is to use a timer but that would really slow down the
424  * transfers if each DMA has to end with a timeout. It would be faster
425  * non-DMA transfers.
426  *
427  * If the device as a very restricted protocol, like just register reads
428  * and writes, then it might possible to implement DMA. However, that
429  * solution would not be general and probably not an appropriate part of
430  * a general OS. But if the interface is unpredictable, such as reading/
431  * variable amounts of data from FLASH, there is more risk. A general
432  * solution might not be possible.
433  */
