70 /* The device structure for radio network device differs from the standard
71  * Ethernet MAC device structure. The main reason for this difference is
72  * that fragmentation must be supported.
73  *
74  * The radio network driver does not use the d_buf packet buffer directly.
75  * Rather, it uses a list smaller frame buffers.
76  *
77  * - Outgoing frame data is provided in an IOB in the via the
78  * r_req_data() interface method each time that the radio needs to
79  * send more data. The length of the frame is provided in the io_len
80  * field of the IOB.
81  *
82  * Outgoing frames are generated when the radio network driver calls
83  * the devif_poll(), devif_timer(), sixlowpan_input(), or
84  * ieee802154_input() interfaces. In each case, the radio driver must
85  * provide a working buffer in the d_buf pointer. A special form of
86  * the packet buffer must be used, struct sixlowpan_reassbuf_s. This
87  * special for includes appended data for managing reassembly of packets.
88  *
89  * - Received frames are provided by radio network driver to the network
90  * via an IOB parameter in the sixlowpan_input() pr ieee802154_input()
91  * interface. The length of the frame is io_len.
92  *
93  * Again, the radio network driver must provide an instance of struct
94  * sixlowpan_reassbuf_s as the packet buffer in the d_buf field. This
95  * driver-provided data will only be used if the the receive frames are
96  * not fragmented.
97  *
98  * - Received 6LoWPAN frames and will be uncompressed and possibly
99  * reassembled in resassembled the d_buf; d_len will hold the size of
100  * the reassembled packet.
101  *
102  * For fagemented frames, d_buf provided by radio driver will not be
103  * used. 6LoWPAN must handle mutliple reassemblies from different
104  * sources simultaneously. To support this, 6LoWPAN will allocate a
105  * unique reassembly buffer for each active reassembly, based on the
106  * reassembly tag and source radio address. These reassembly buffers
107  * are managed entirely by the 6LoWPAN layer.
108  *
109  * This is accomplished by "inheriting" the standard 'struct net_driver_s'
110  * and appending the frame buffer as well as other metadata needed to
111  * manage the fragmentation. 'struct radio_driver_s' is cast
112  * compatible with 'struct net_driver_s' when dev->d_lltype ==
113  * NET_LL_IEEE802154 or dev->d_lltype == NET_LL_PKTRADIO.
114  *
115  * The radio network driver has reponsibility for initializing this
116  * structure. In general, all fields must be set to NULL. In addition:
117  *
118  * 1. On a TX poll, the radio network driver should provide its driver
119  * structure along is (single) reassemby buffer provided at d_buf.
120  * During the course of the poll, the networking layer may generate
121  * outgoing frames. These frames will by provided to the radio network
122  * driver via the req_data() method.
123  *
124  * After sending each frame through the radio, the MAC driver must
125  * return the frame to the pool of free IOBs using the iob_free().
126  *
127  * 2. When receiving data both buffers must be provided for 6LoWPAN
128  * frames; PF_IEEE802154 frames do not require the d_buf.
129  *
130  * The radio driver should receive the frame data directly into the
131  * payload area of an IOB frame structure. That IOB structure may be
132  * obtained using the iob_alloc() function.
133  *
134  * For 6LoWPAN, fragmented packets will be reassembled using allocated
135  * reassembly buffers that are managed by the 6LoWPAN layer. The radio
136  * driver must still provide its (single) reassembly buffer in d_buf;
137  * that buffer is still used for the case where the packet is not
138  * fragmented into many frames. In either case, the packet buffer will
139  * have a size of advertised MTU of the protocol, CONFIG_NET_6LOWPAN_MTU,
140  * plus CONFIG_NET_GUARDSIZE and some additional overhead for reassembly
141  * state data.
142  *
143  * The radio network driver should then inform the network of the recipt
144  * of a frame by calling sixlowpan_input() or ieee802154_input(). That
145  * single frame (or, perhaps, list of frames) should be provided as
146  * second argument of that call.
147  *
148  * The network will free the IOB by calling iob_free after it has
149  * processed the incoming frame. As a complexity, the result of
150  * receiving a frame may be that the network may respond provide an
151  * outgoing frames in the via a nested call to the req_data() method.
152  */
153
