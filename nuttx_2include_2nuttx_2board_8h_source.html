35 /* This header file contains function prototypes for the interfaces between
36  * (1) the nuttx core-code, (2) the microprocessor specific logic that
37  * resides under the arch/ sub-directory, and (3) the board-specific logic
38  * that resides under configs/
39  *
40  * Naming conventions:
41  *
42  * 1. Common Microprocessor Interfaces.
43  *
44  * Any interface that is common across all microprocessors should be
45  * prefixed with up_ and prototyped in this header file. These
46  * definitions provide the common interface between NuttX and the
47  * architecture-specific implementation in arch/
48  *
49  * These definitions are retained in the header file
50  * nuttx/include/arch.h
51  *
52  * NOTE: up_ is supposed to stand for microprocessor; the u is like the
53  * Greek letter micron: µ. So it would be µP which is a common shortening
54  * of the word microprocessor.
55  *
56  * 2. Microprocessor-Specific Interfaces.
57  *
58  * An interface which is unique to a certain microprocessor should be
59  * prefixed with the name of the microprocessor, for example stm32_,
60  * and be prototyped in some header file in the arch/ directories.
61  *
62  * There is also a arch/<architecture>/include/<chip>/chip.h header file
63  * that can be used to communicate other microprocessor-specific
64  * information between the board logic and even application logic.
65  * Application logic may, for example, need to know specific capabilities
66  * of the chip. Prototypes in that chip.h header file should follow the
67  * microprocessor specific naming convention.
68  *
69  * 3. Common Board Interfaces.
70  *
71  * Any interface that is common across all boards should be prefixed
72  * with board_ and should be prototyped in this header file. These
73  * board_ definitions provide the interface between the board-level
74  * logic and the architecture-specific logic.
75  *
76  * Board related declarations are retained in this header file.
77  *
78  * There is also a configs/<board>/include/board.h header file that
79  * can be used to communicate other board-specific information between
80  * the architecture logic and even application logic. Any definitions
81  * which are common between a single architecture and several boards
82  * should go in this board.h header file; this file is reserved for
83  * board-related definitions common to all architectures.
84  *
85  * 4. Board-Specific Interfaces.
86  *
87  * Any interface which is unique to a board should be prefixed with
88  * the board name, for example stm32f4discovery_. Sometimes the board
89  * name is too long so stm32_ would be okay too. These should be
90  * prototyped in configs/<board>/src/<board>.h and should not be used
91  * outside of that board directory since board-specific definitions
92  * have no meaning outside of the board directory.
93  */

643  * Description:
644  * If CONFIG_BOARD_CRASHDUMP is selected then up_asseert will call out to
645  * board_crashdump prior to calling exit in the case of an assertion failure.
646  * Or in the case of a hardfault looping indefinitely. board_crashdump then
647  * has a chance to save the state of the machine. The provided
648  * board_crashdump should save as much information as it can about the cause
649  * of the fault and then most likely reset the system.
650  *
651  * N.B. There are limited system resources that can be used by the provided
652  * board_crashdump function. The tems from the fact that most critical/fatal
653  * crashes are because of a hard fault or during interrupt processing.
654  * In these cases, up_assert is running from the context of an interrupt
655  * handlerand it is impossible to use any device driver in this context.
656  *
657  * Also consider the following: Who knows what state the system is in? Is
658  * memory trashed? Is the Heap intact? Therefore all we can expect to do in
659  * board_crashdump is save the "machine state" in a place where on the next
660  * reset we can write it to more sophisticated storage in a sane operating
661  * environment.
662  *
663  ****************************************************************************/
