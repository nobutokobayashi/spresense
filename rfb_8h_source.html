56 /* 6.1.1 ProtocolVersion
57  *
58  * "Handshaking begins by the server sending the client a ProtocolVersion
59  * message. This lets the client know which is the highest RFB protocol
60  * version number supported by the server. The client then replies with a
61  * similar message giving the version number of the protocol which should
62  * actually be used (which may be different to that quoted by the server).
63  * A client should never request a protocol version higher than that
64  * offered by the server. It is intended that both clients and servers may
65  * provide some level of backwards compatibility by this mechanism.
66  *
67  * "The only published protocol versions at this time are 3.3, 3.7, 3.8
68  * (version 3.5 was wrongly reported by some clients, but this should be
69  * interpreted by all servers as 3.3). Addition of a new encoding or
70  * pseudo-encoding type does not require a change in protocol version,
71  * since a server can simply ignore encodings it does not understand.
72  *
73  * "The ProtocolVersion message consists of 12 bytes interpreted as a string
74  * of ASCII characters in the format "RFB xxx.yyy\n" where xxx and yyy are
75  * the major and minor version numbers, padded with zeros."
76  */
77

285 /* "Server-pixel-format specifies the server’s natural pixel format. This
286  * pixel format will be used unless the client requests a different format
287  * using the SetPixelFormat message (section 6.4.1).
288  *
289  * "Bits-per-pixel is the number of bits used for each pixel value on the
290  * wire. This must be greater than or equal to the depth which is the number
291  * of useful bits in the pixel value. Currently bits-per-pixel must be 8,
292  * 16 or 32—less than 8-bit pixels are not yet supported. Big-endian-flag is
293  * non-zero (true) if multi-byte pixels are interpreted as big endian. Of
294  * course this is meaningless for 8 bits-per-pixel.
295  *
296  * "If true-colour-flag is non-zero (true) then the last six items specify
297  * how to extract the red, green and blue intensities from the pixel value.
298  * Red-max is the maximum red value (= 2n − 1 where n is the number of bits
299  * used for red). Note this value is always in big endian order. Red-shift
300  * is the number of shifts needed to get the red value in a pixel to the
301  * least significant bit. Green-max, green-shift and blue-max, blue-shift
302  * are similar for green and blue. For example, to find the red value
303  * (between 0 and red-max) from a given pixel, do the following:
304  *
305  * - "Swap the pixel value according to big-endian-flag (e.g. if
306  * big-endian-flag is zero (false) and host byte order is big endian,
307  * then swap).
308  * - "Shift right by red-shift.
309  * - "AND with red-max (in host byte order).
310  *
311  * "If true-colour-flag is zero (false) then the server uses pixel values
312  * which are not directly composed from the red, green and blue intensities,
313  * but which serve as indices into a colour map. Entries in the colour map
314  * are set by the server using the SetColourMapEntries message (section
315  * 6.5.2)."
316  */

381 /* 6.4.3 FramebufferUpdateRequest
382  *
383  * "Notifies the server that the client is interested in the area of the
384  * framebuffer specified by x-position, y-position, width and height. The
385  * server usually responds to a FramebufferUpdateRequest by sending a
386  * FramebufferUpdate. Note however that a single FramebufferUpdate may be
387  * sent in reply to several FramebufferUpdateRequests.
388  *
389  * "The server assumes that the client keeps a copy of all parts of the
390  * framebuffer in which it is interested. This means that normally the
391  * server only needs to send incremental updates to the client.
392  *
393  * "However, if for some reason the client has lost the contents of a
394  * particular area which it needs, then the client sends a
395  * FramebufferUpdateRequest with incremental set to zero (false). This
396  * requests that the server send the entire contents of the specified area
397  * as soon as possible. The area will not be updated using the CopyRect
398  * encoding.
399  *
400  * "If the client has not lost any contents of the area in which it is
401  * interested, then it sends a FramebufferUpdateRequest with incremental
402  * set to non-zero (true). If and when there are changes to the specified
403  * area of the framebuffer, the server will send a FramebufferUpdate. Note
404  * that there may be an indefinite period between the
405  * FramebufferUpdateRequest and the FramebufferUpdate.
406  *
407  * "In the case of a fast client, the client may want to regulate the rate
408  * at which it sends incremental FramebufferUpdateRequests to avoid hogging
409  * the network."
410  */

416  uint8_t xpos[2]; /* U16 X-position */
417  uint8_t ypos[2]; /* U16 Y-position */
418  uint8_t width[2]; /* U16 Width */
419  uint8_t height[2]; /* U16 Height */
420 };

437 /* "The interpretation of keysyms is a complex area. In order to be as
438  * widely interoperable as possible the following guidelines should be
439  * used:
440  *
441  * - "The 'shift state' (i.e. whether either of the Shift keysyms are down)
442  * should only be used as a hint when interpreting a keysym. For
443  * example, on a US keyboard the ’#’ character is shifted, but on a UK
444  * keyboard it is not. A server with a US keyboard receiving a ’#’
445  * character from a client with a UK keyboard will not have been sent
446  * any shift presses. In this case, it is likely that the server will
447  * internally need to 'fake' a shift press on its local system, in order
448  * to get a ’#’ character and not, for example, a ’3’.
449  * - "The difference between upper and lower case keysyms is significant.
450  * This is unlike some of the keyboard processing in the X Window System
451  * which treats them as the same. For example, a server receiving an
452  * uppercase ’A’ keysym without any shift presses should interpret it
453  * as an uppercase ’A’. Again this may involve an internal 'fake' shift
454  * press.
455  * - "Servers should ignore 'lock' keysyms such as CapsLock and NumLock
456  * where possible. Instead they should interpret each character-based
457  * keysym according to its case.
458  * - "Unlike Shift, the state of modifier keys such as Control and Alt
459  * should be taken as modifying the interpretation of other keysyms.
460  * Note that there are no keysyms for ASCII control characters such
461  * as ctrl-a - these should be generated by viewers sending a Control
462  * press followed by an ’a’ press.
463  * - "On a viewer where modifiers like Control and Alt can also be used to
464  * generate character-based keysyms, the viewer may need to send extra
465  * 'release' events in order that the keysym is interpreted correctly.
466  * For example, on a German PC keyboard, ctrl-alt-q generates the ’@’
467  * character. In this case, the viewer needs to send 'fake' release
468  * events for Control and Alt in order that the ’@’ character is
469  * interpreted correctly (ctrl-alt-@is likely to mean something
470  * completely different to the server).
471  * - "There is no universal standard for 'backward tab' in the X Window
472  * System. On some systems shift+tab gives the keysym 'ISO Left Tab',
473  * on others it gives a private 'BackTab' keysym and on others it gives
474  * 'Tab' and applications tell from the shift state that it means
475  * backward-tab rather than forward-tab. In the RFB protocol the latter
476  * approach is preferred. Viewers should generate a shifted Tab rather
477  * than ISO Left Tab. However, to be backwards-compatible with existing
478  * viewers, servers should also recognise ISO Left Tab as meaning a
479  * shifted Tab."
480  */
